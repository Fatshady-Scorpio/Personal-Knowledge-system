## 一、缘起：为什么做这个项目

### 1.1 知识管理的三大痛点

作为一名关注 AI 领域的产品经理，日常面临三个知识管理难题：

```
  ┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
  │   信息过载        │     │   碎片化存储      │     │   检索效率低      │
  │                  │     │                  │     │                  │
  │ 每天接触大量内容  │     │ 浏览器 / Notion  │     │ 需要回顾时       │
  │ 但读后即忘       │     │ 本地笔记 / 收藏   │     │ 要么翻找笔记      │
  │ 无法形成积累      │     │ 散落在多个系统    │     │ 要么重新搜索      │
  └──────────────────┘     └──────────────────┘     └──────────────────┘
```

市面上所有知识管理工具的共性问题是：

> **只解决「存」，不解决「用」和「长」。**

### 1.2 Karpathy 的启发

2025 年，Andrej Karpathy 提出了 **"LLM Wiki"** 的概念。核心思想非常简单但颠覆：

> 不追求自动采集互联网内容，而是人工精选 Raw 材料，用 LLM 将 Raw 材料编译成结构化的 Wiki 词条，词条之间通过双向链接互联。随着时间推移，Wiki 会自动积累、生长，形成**知识复利**。

### 1.3 两种路线的本质区别

| 维度 | 传统 RAG | LLM Wiki（Agentic Wiki） |
|:---:|:---|:---|
| **内容来源** | 自动爬取，量大 | 人工精选，量小但精 |
| **存储方式** | 向量数据库（无结构） | 结构化 Markdown + 双向链接 |
| **知识积累** | 线性堆积 | 网状复利（链接产生新关联） |
| **查询方式** | 无状态语义检索 | 有状态 Wiki 导航 |
| **可维护性** | 难以人工干预 | 人工可阅读、可编辑、可审核 |
| **知识形态** | 黑盒向量 | 人类可读的词条网络 |

**核心决策**：选择 Agentic Wiki 路线 -- 让知识系统像维基百科一样有结构、可浏览、可维护，而不是一个黑盒向量库。

### 1.4 核心范式对比

```
        传统 RAG                          Agentic Wiki
    ┌───────────────┐                 ┌───────────────────┐
    │  爬取所有内容   │                 │  人工精选 Raw 材料 │
    │  ┌───────────┐ │                 │  ┌─────────────┐  │
    │  │ 向量存储   │ │                 │  │ LLM 编译     │  │
    │  │ (黑盒)     │ │                 │  │ (可审核)     │  │
    │  └───────────┘ │                 │  └──────┬───────┘  │
    │       │        │                 │         │          │
    │  语义相似度检索 │                 │  ┌──────▼───────┐  │
    │  (无解释)      │                 │  │ Wiki 词条网络 │  │
    └───────────────┘                 │  │ (双向链接)    │  │
                                      │  └──────┬───────┘  │
                                      │         │          │
                                      │  ┌──────▼───────┐  │
                                      │  │ 有状态查询    │  │
                                      │  │ (知识导航)    │  │
                                      │  └──────────────┘  │
                                      └───────────────────┘
```

## 二、演进历程：从概念到落地的四步

### 阶段一：探索与试错 -- RAG 模式

项目最早期采用了传统 RAG 路线，构建了一个全自动化的知识采集系统：

```
RSS/爬虫采集 ──► 优先级过滤 ──► 摘要生成 ──► 向量存储 ──► 语义搜索/RAG问答
    │              │               │             │              │
    ▼              ▼               ▼             ▼              ▼
 50+信源        关键词打分       LLM摘要      FAISS+BM25     回答生成
 网页爬取       时效加成         分类标签      知识图谱        日报周报
 视频字幕       忽略词过滤       文件保存      NetworkX       APScheduler
```

**包含的模块（共 13 个目录，50+ 文件）：**

| 模块 | 文件数 | 功能 |
|:---|:---:|:---|
| `src/collector/` | 4 | RSS 抓取、网页爬虫、视频字幕提取 |
| `src/retriever/` | 8 | 向量存储、BM25、语义搜索、RAG 问答 |
| `src/processor/` | 4 | 摘要生成、优先级过滤、日报、周报 |
| `src/scheduler/` | 3 | APScheduler 定时任务调度 |
| `src/graph/` | 3 | SQLite + NetworkX 知识图谱 |
| `src/agent/` | 3 | 旧 RAG API、Context Injector |
| `src/mcp/` | 2 | MCP Server 工具定义 |

**发现的五个核心问题：**

| # | 问题 | 影响 |
|:---:|:---|:---|
| 1 | **自动采集 = 信息垃圾场** | RSS/爬虫灌入大量低质量噪声，知识库线性堆积 |
| 2 | **向量搜索是黑盒** | 用户不知道"为什么返回这个结果"，无法干预 |
| 3 | **自动摘要质量不可控** | LLM 自动摘要无法替代人工筛选后的精炼输入 |
| 4 | **自动化调度 = 成本浪费** | 定时任务持续消耗 API 额度，产出价值不高 |
| 5 | **没有知识复利** | 每篇文章孤立存储，没有链接产生新关联 |

> **关键结论**：自动化采集这条路走不通。知识的价值不在于「量」，而在于「精选」和「关联」。

### 阶段二：架构重构 -- 转向 Agentic Wiki

基于阶段一的教训，进行了根本性的架构重构：

**五项设计原则：**

| # | 原则 | 说明 |
|:---:|:---|:---|
| 1 | **手动输入，不可变** | Raw 材料由用户手动保存，系统不修改原始内容 |
| 2 | **LLM 编译，可审核** | Raw → Wiki 的过程透明，每个词条可人工审核 |
| 3 | **双向链接，网状生长** | `[[概念]]` 语法互联，形成知识网络 |
| 4 | **index.md 导航** | LLM 生成层级索引，人类 + LLM 双入口 |
| 5 | **手动触发** | 编译、查询、健康检查全由用户控制 |

**新架构的数据流：**

```
┌─────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   raw/      │     │   compiler/       │     │     wiki/        │
│  Raw 材料    │────►│  LLM 编译器       │────►│  Wiki 词条网络    │
│ (手动输入)   │     │  概念提取         │     │  (结构化知识)     │
│             │     │  双向链接         │     │  index.md        │
└─────────────┘     └──────────────────┘     └────────┬─────────┘
                                                      │
                                        ┌─────────────▼──────────┐
                                        │      query/            │
                                        │   有状态深度查询        │
                                        │   Token 预算管理        │
                                        │   链接遍历              │
                                        └─────────────┬──────────┘
                                                      │
                                        ┌─────────────▼──────────┐
                                        │   maintenance/         │
                                        │   健康检查              │
                                        │   矛盾/过期/断裂检测    │
                                        └────────────────────────┘
```

## 三、当前系统：完整的 Agentic Wiki

### 3.1 核心工作流：四步闭环

```
  ┌──────────────┐
  │  第一步       │
  │  手动输入     │ ─────────────────────────────────────┐
  │  Raw 材料     │                                      │
  └──────┬───────┘                                      │
         │                                              ▼
  ┌──────▼───────┐     ┌──────────────┐          ┌──────────────┐
  │  第二步       │     │  第四步       │          │  第三步       │
  │  LLM 编译     │────►│  健康检查     │◄─────────│  深度查询     │
  │  Wiki 词条    │     │  维护清理     │          │  知识复利     │
  └──────────────┘     └──────────────┘          └──────────────┘
```

### 3.2 第一步：手动输入 Raw 材料

用户将自己筛选的有价值内容保存为 Markdown 文件：

```markdown
---
type: article
source: https://example.com/article
tags: [LLM, Transformer]
status: raw
---

# 文章标题

[内容摘要或关键段落]

---

## 我的思考

[手写思考 / 个人理解] ← 这是知识区别于信息的核心
```

**三种输入方式：**

| 方式 | 场景 | 工具 |
|:---|:---|:---|
| 手动创建 Markdown | 最常用，直接粘贴内容 | 任意编辑器 |
| PDF 转 Markdown | 论文、报告等 PDF 文档 | `scripts/pdf_to_md.py` |
| 英文内容直接粘贴 | 外文论文、技术博客 | 系统自动翻译编译 |

### 3.3 第二步：LLM 编译（Raw → Wiki）

```bash
PYTHONPATH=. python scripts/compile_raw.py --all
```

编译流程的 7 个步骤：

```
  1. 读取 Raw 材料
       │
       ▼
  2. 检测内容语言 ──► 中文 ──► 使用中文提取 prompt
       │               │
       │               └──► 英文 ──► 使用双语提取 prompt (含翻译)
       ▼
  3. LLM 提取 3-8 个核心概念 (JSON 格式)
       │
       ▼
  4. 为每个概念生成结构化词条
       │   ├── 定义（50 字以内）
       │   ├── 详细解释（200-300 字）
       │   ├── 关键点（3-5 条）
       │   └── 相关概念（双向链接）
       ▼
  5. 创建 Wiki 词条文件 (wiki/concepts/)
       │
       ▼
  6. 更新 wiki/index.md 导航（增量更新）
       │
       ▼
  7. 将 Raw 材料状态标记为 "compiled"
```

**编译后的 Wiki 词条示例：**

```markdown
---
type: concept
created_from: raw/articles/xxx.md
created_at: 2026-04-15T10:30:00
related_topics: [[Transformer]], [[Attention Mechanism]]
confidence: 0.92
tags: [LLM, AI]
---

# Transformer 架构

## 定义
Transformer 是一种基于自注意力机制的神经网络架构...

## 详解
[详细解释 200-300 字]

## 关键要点
- 核心是自注意力机制
- 支持并行训练
- 已成为 LLM 的标准架构

## 相关概念
[[注意力机制]]  [[位置编码]]  [[多头注意力]]

## 来源
- [[文章标题]]
```

### 3.4 第三步：深度查询（有状态 Wiki 导航）

```bash
PYTHONPATH=. python scripts/chat.py   # 交互式聊天
```

查询流程详解：

```
  用户提问："Transformer 和 RNN 的区别？"
       │
       ▼
  ┌─────────────────────────────────┐
  │ Step 1: 从 index.md 加载相关索引  │
  │         匹配包含 "Transformer"    │
  │         和 "RNN" 的主题段落       │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ Step 2: 加载匹配的 Wiki 词条      │
  │         读取 wiki/concepts/       │
  │         下的完整词条内容           │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ Step 3: Token 预算管理           │
  │         控制在预算范围内           │
  │         (默认 100,000 tokens)    │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ Step 4: 双向链接遍历 (1-hop)     │
  │         从已加载词条出发           │
  │         找到所有 [[关联]] 词条     │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ Step 5: 查找已有 Q&A 记录         │
  │         复用之前的高质量问答        │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ Step 6: LLM 综合生成回答          │
  │         标注知识边界              │
  │         (Wiki 知识 vs 通用知识)   │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ Step 7: 保存高质量 Q&A            │
  │         自动存入 outputs/qa/      │
  │         供下次查询复用             │
  └─────────────────────────────────┘
```

**回答示例：**

```
### 来自知识库的内容

根据知识库中的 [[Transformer 架构]] 词条...

### 补充说明

知识库尚未收录 RNN 的详细对比，但一般来说...
```

### 3.5 第四步：健康检查

```bash
PYTHONPATH=. python scripts/health_check.py --verbose
```

| 检查项 | 检测内容 | 处理方式 |
|:---|:---|:---|
| **断裂链接** | `[[概念]]` 但词条不存在 | 需要创建缺失的词条 |
| **孤岛词条** | 没有任何入链或出链 | 需要添加链接或删除 |
| **潜在矛盾** | LLM 两两比对词条内容 | 需要人工审核 |
| **过期内容** | 超过 90 天未更新 | 需要更新或归档 |
| **来源缺失** | 词条没有来源元数据 | 需要补充引用 |

### 3.7 技术栈

| 层面 | 技术选择 | 理由 |
|:---|:---|:---|
| **LLM 调用** | 阿里云百炼 API（Anthropic 兼容） | 支持多模型路由，8+ 模型可选 |
| **模型选择** | 通义 3.6 Plus（默认）<br>GLM-5（降级） | 中文能力强<br>中文能力强，成本低 |
| **存储格式** | Markdown 文件 + 双向链接 | 人类可读、可编辑、Git 版本控制 |
| **Web 框架** | FastAPI + uvicorn | 轻量、高性能、异步 |
| **Token 计数** | tiktoken | 计数准确，支持多种 encoding |
| **项目结构** | Python 模块化<br>compiler / query / maintenance | 清晰的职责分离<br>高内聚低耦合 |

## 四、关键设计决策

### 4.1 为什么不用向量数据库？

向量数据库（FAISS / Pinecone / Weaviate）的优势是大规模语义检索，但对我们的场景有四个不匹配：

```
              向量数据库方案                      Markdown Wiki 方案
         ┌─────────────────────┐            ┌─────────────────────────┐
         │  ● ● ● ● ●           │            │  [[概念A]] ──► [[概念B]] │
         │  ● ● ● ●             │            │      │           │       │
         │  ● ● ●               │            │      ▼           ▼       │
         │  (孤立点云)           │            │  [[概念C]] ◄── [[概念D]] │
         │                     │            │  (互联网络)              │
         │  可解释性：差         │            │  可解释性：完全透明        │
         │  可维护性：无法编辑    │            │  可维护性：任意编辑        │
         └─────────────────────┘            └─────────────────────────┘
```

| 维度 | 向量数据库 | Markdown Wiki |
|:---|:---|:---|
| 规模适配 | 适合百万级文档 | 适合几十到几百级 |
| 可解释性 | 差（不知道"为什么返回"） | 完全透明（链接即路径） |
| 可维护性 | 无法直接编辑 | 任意文本编辑器可改 |
| 知识关联 | 孤立存储 | 双向链接网络 |
| 版本控制 | 不支持 | Git 天然支持 |

**我们的方案**：`index.md` 导航 + 双向链接图。LLM 先通过索引定位相关主题，再遍历链接获取上下文。完全可解释、可编辑、可维护。

### 4.2 为什么手动输入而不是自动采集？

这是从旧架构踩坑后得出的最重要结论：

```
         自动采集流程                          手动输入流程
    ┌───────────────────┐                ┌───────────────────┐
    │ RSS: 每天 50 篇    │                │ 用户主动保存:        │
    │ 有价值: 5 篇 (10%) │                │ 每天 1-2 篇        │
    │ 噪声:  45 篇 (90%) │                │ 价值率: ~90%       │
    │ 价值密度: 极低       │                │ 价值密度: 高        │
    └───────────────────┘                └───────────────────┘
              │                                     │
              ▼                                     ▼
    ┌───────────────────┐                ┌───────────────────┐
    │  知识库变成         │                │  每条知识都经过     │
    │  "信息垃圾场"       │                │  用户思考和筛选     │
    └───────────────────┘                └───────────────────┘
```

- **自动采集 = 低质量噪声灌入**：RSS 和爬虫每天灌入 10-50 篇内容，但真正有价值的不到 5%
- **人工筛选 = 高质量输入**：用户主动保存的内容，本身就是经过思考的精选
- **用户思考是知识的核心**：Raw 材料中的「我的思考」部分，是知识区别于信息的关键

### 4.3 为什么用 Markdown 而不是数据库？

| 优势 | 说明 |
|:---|:---|
| **Obsidian 原生兼容** | 直接浏览、编辑、可视化知识图谱，零成本接入 |
| **Git 友好** | 每次编译和修改都可通过 Git 追踪知识生长历史 |
| **人类可读** | 不需要任何工具就能打开阅读，降低使用门槛 |
| **LLM 友好** | LLM 天然理解 Markdown 格式，不需要额外解析层 |

### 4.4 多模型路由的意义

系统支持 8+ 模型，按任务自动路由，并配备完整的容错机制：

```
                    任务请求
                      │
          ┌───────────┼───────────┐
          │           │           │
          ▼           ▼           ▼
    ┌───────────┐ ┌──────────┐ ┌──────────┐
    │ 概念提取   │ │  翻译     │ │ 查询回答  │
    │           │ │          │ │          │
    ▼           ▼ ▼          ▼ ▼          ▼
  qwen3.6   qwen3.6-plus  qwen3.6-plus
   -plus     (快速响应)    (综合能力)
    │           │           │
    └───────────┼───────────┘
                │
                ▼
      ┌─────────────────┐
      │  容错机制        │
      │  ┌───────────┐  │
      │  │ 5 次重试   │  │
      │  │ 指数退避   │  │
      │  │ 自动降级   │──┼──► GLM-5 (备用)
      │  │ 到 GLM-5  │  │
      │  └───────────┘  │
      └─────────────────┘
```

**容错保障**：即使主模型不可用，服务也不会中断。5 次重试 + 指数退避 + 自动降级到 GLM-5。
